Dms Life Cycle

If using this Life Cycle, EDNAR will interact with the DMS to create either a Switching Schedule or a job in the DMS, depending on configuration.

If a DMS schedule is created, EDNAR will track the approval state of that Schedule in the DMS.

If a DMS job is created, there are no approval states in the DMS.

For this life cycle, a NAR only ever goes to the Accepted state if there has been a failure to create a schedule or job in the DMS. If everything is working correctly, it normally transitions from Submitted to DMS Created when a user accepts the NAR. A user can attempt to re-send the NAR to the DMS when it is in the Accepted state (there is a button for this action), it does not have to be “re-submitted”.

If the Schedule or Job is not created, the user will be told with a pop-up error message, and it will also be logged as an exception against the NAR. A report can be set up showing all NARs that have not been sent to the DMS, which is useful for managing an issue where the DMS integration may have been down for an extended period. Hopefully, this will rarely happen.

There is no automatic message sent to the user to say the DMS integration is active again, but an admin can send a broadcast message to all users to let them know the DMS is available again.

A NAR can be modified after it has been sent to the DMS – for instance check list items. And any fields that have been mapped to the DMS schedule header will be updated in the DMS when they change in the NAR. Some applicants may also have permissions that allow these edits after being accepted.
