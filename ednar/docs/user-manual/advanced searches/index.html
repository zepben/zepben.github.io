Advanced Searches

The advanced NAR search functionality enables users to create custom filtering rules to precisely tailor their NAR search results. By defining specific criteria, users can narrow down the displayed NARs to those that meet their exact requirements.

Unlike the basic search input screen, the fields that are used in the filter are added to a query.

Creating an Advanced Search starts in a similar way to creating a basic search, but the way the search is built looks very different, as illustrated below.

Searches are created by specifying a set of rules that consist of the following components:

  • Field: Specifies the attribute of the NAR to be compared. For example, the ‚ÄúSwitching Type‚Äù.
  • Operator: Determines the type of comparison to be performed on the selected field. For example ‚ÄúIs equal to‚Äù.
  • Value: Defines the value against which the field will be compared, for example ‚ÄúTrue‚Äù or ‚ÄúFalse‚Äù
  • Group: Rules (and sub-groups) are connected using an AND or OR connector, enforcing different logical conditions for filtering. Examples are provided below.

Building a rule set

  1. Upon opening the builder, a top-level group is displayed. This group serves as the container for all rules and sub-groups defined by the custom filter. The rules and sub-groups can be made to follow ‚ÄúAND‚Äù or ‚ÄúOR‚Äù Boolean logic.

  1. Add a Rule: Click the 'Add Rule' button to insert a new rule into the group.

  1. Select Field: Choose the attribute you wish to compare against in your rule. An attribute can be related directly to the NAR, such as the NAR State, as illustrated below. Note how the incremental search is picking up all attributes with the letters NAR in them.

An attribute can also be associated with Checklist Questions. In the example below, a Planner Question (which exists in the Planners Checklist) has been selected as an attribute. When this happens, an additional field selector is provided to select which question is to be used.

In some cases, the field selected may be data that can be held against one or more authorities or customer notification tasks. For example, it is possible to search all NARs that have a specified NMI, or a specified isolation point. In these cases, the search will consider all authorities and all customer notification tasks to find the NARs with this data.

  1. Select Operator: Specify the comparison operator to be applied for this rule.

  1. Select Value: Define the value against which the field will be compared using the selected operator. Depending on the type of field, the operator and type selectable as the comparison value will be different, as illustrated in the two examples below where a date attribute and TRUE/FALSE attribute have been picked.

  1. Refine Your Filter: After defining a rule, users can either apply the filter by clicking the 'Search' button or continue expanding the filter by adding more rules and groups.

Understanding Connectors:

  1. Each group features a connector that determines the logical relationship between its contained rules and sub-groups.

  • Use the AND connector to require all linked rules and groups to be satisfied.

  • Use the OR connector to necessitate at least one of the linked rules or groups to be satisfied.

  1. Create additional groups as needed.
