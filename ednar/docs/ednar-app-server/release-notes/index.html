[6.8.0]​

Breaking Changes​

  • The ASSET_STRING_FORMAT app option now uses an updated mustache template to support specific equipment containers. The following variables are supported:
    • assetId
    • voltage
    • circuits
    • feeders
    • lvFeeders
    • substations
    • sites
    • energizingFeeders
    • energizingSubstation
    • poiText
    • assetName

New Features​

  • Added support for configuring NAR transition reasons based on JobType.
  • Introduced a new GraphQL query: getNarTransitionReasons(jobTypeId: Long?, type: NarTransitionReasonType?, limit: Int?)
    • If jobTypeId is provided, returns transition reasons configured for that specific job type. If not provided, returns all transition reasons from the database.
    • If type is specified, filters the results by the given NarTransitionReasonType.
    • If limit is specified, restricts the number of results returned.
  • Added getUpstreamTraceEquipment and getDownstreamTraceEquipment query to support additional tracing capabilities.
  • Added new GraphQL Api getMapSymbolDetails() to query list of available map symbols details.
  • Added new route and service (EwbNetworkCustomerServiceUtils) to fetch customer info by mRIDs via GraphQL API call to EWB.
  • Added support for custom messages in customer notification tasks with delivery method templates and audit logging.
  • Added findClashNmisByNarId() GraphQL endpoint to check if requested NMIs exist in other NARs within a given nar start and end date time and return the corresponding NAR IDs.
  • Added a permission based access control for viewing customer sensitive information in the customer list table in the UI.
    • Added the following GraphQl queries to configure the customer list columns:
      • getUserSavedCustomerListColumns()
      • getUserPermittedCustomerListColumns.
    • Added the following GraphQL mutation to save the user configured customer list columns:
      • saveUserCustomerListColumns()
    • Added a new REST endpoint, /log-customer-data to log accessed customer information, including the NMIs viewed and the date and time of access.
    • The following privileges are required to view customer sensitive (PII) data:
      • READ_CUSTOMER_PII
      • EXPORT_CUSTOMER_PII
  • Added new routes to fetch location points for green-circle rendering and retrieve customer info by supply point.
  • Added support to query metrics field in isolationTrace query, which will return the network metrics summary from the trace results, such as conductor lengths, number of connected customers, and connected transformer KVA.
  • Added new callouts table to separately store callouts on all types of annotations.
  • Added new annotation_colours table to allow configuration of colours available for annotations in the front-end. This does not modify existing annotations, only configures the palette available to users when creating new ones.
  • Added new getAnnotationColours GraphQL query to allow for these colours to be retrieved by the front-end.
  • Added label to lv_switching_steps to enable storage of string labels against switching steps, and the same property to both GraphQL queries and REST endpoints that serialise switching steps.
  • Added default_callout_colour to lv_switching_actions to enable a default colour to be set for an action, and the same property to both GraphQL queries and REST endpoints that serialise switching actions.
  • Added new DEFAULT_CALLOUT_WIDTH and DEFAULT_CALLOUT_HEIGHT app options to define front-end behaviour when the user creates new callouts.
  • Added new CALLOUT_MINIMUM_ZOOM app option to define front-end behaviour for when a user views callouts on the map.
  • Added new SWITCHING_STEP_CALLOUT_FORMAT app option to determine how callout descriptions are auto-populated when created from switching steps.
  • Add support for customer notification tasks having a "parent task"

Enhancements​

  • Add new JSON configuration option http.disableGraphQlIntrospection (default false) to disable GraphQL introspection queries for improved security. __typename field selection on ordinary GraphQL queries will be allowed in either case.
  • Add new JSON configuration option http.graphQlQueryDepthLimit (default none). If set, incoming GraphQL queries are analysed for their maximum depth and rejected if they exceed this limit.
  • Add new JSON configuration option http.useSecureCookies to force use or non-use of the Secure flag when sending Set-Cookie headers in responses. Defaults to whether https is set or not. Useful for running EDNAR behind a reverse proxy.
  • Add new JSON configuration option http.corsAllowedOrigins to specify a list of allowed origins for CORS requests. If not set, Access-Control-Allow-Origin headers will never be sent. If one of the options is set to *, all origins will be allowed.
  • Added support for tracking removed NMIs by introducing the following columns in the customer_notification_nmis table:
    • removedBy: The user who mark NMI as removed.
    • removeAt : The timestamp when then NMI is marked as removed.
  • New LV_SWITCHING_ASSET_SEARCH_ENABLED app option added.
  • Added energizing feeders and energizing substation to the response of the find-assets endpoint.
  • Added a new AccessAuthorityTypeField ISOLATION_TRACE_SUMMARY that can be configured in the authority_types_fields_config table. If no value is set, it’s visible; otherwise, it can be set to HIDDEN.
  • Enhanced customer notification type mapping to also support mapping by delivery method.
    • Added new triggers_outbound_request flag to control whether an event should send an outbound.
    • Added "eic": [] field to outbound payload for EIC details i-e EIC_DELIVERED.
  • In both GraphQL and the REST API, annotation has been separated into two components, geoJson and callouts, with callouts having a defined data structure.
  • Saving a NAR now always creates an audit log entry, even in Draft.
  • New notification action triggers added for customer notification events.
  • New CUSTOMER_NOTIFICATION_TASK_PARENT_TASK_ID and COMPLETION_COMMENTS NAR Search columns.
  • (LTS 6.6.X) Added new QUICK_SEARCH_OPENS_IN_NEW_TAB app option. Used by frontend.
  • (LTS 4.35.X) Customer outage overlay requests are now non-blocking.
  • (LTS 4.35.X) A new app option called CUSTOMER_OUTAGE_OVERLAY_FILTER_MAYBE_TASKS. If set to true, NMIs on tasks with an outage type of MAYBE are not sent to the Customer Outage Overlay Service.

Fixes​

  • SameSite attribute is now correctly set to Strict on all cookies set by the server.
  • Updated GetWebhooksByActionQuery to avoid Oracle error with a SELECT DISTINCT query with a CLOB column included.
  • Setting the user property in the database connection configuration will no longer cause the currentSchema to be set to the username when using a PostgreSQL database.
  • In REST endpoints, annotation is now consistently used across all objects instead of annotations.
  • (LTS 6.6.X) Fixed capitalisation in JSON serialization for mRID attribute on network hierarchy DTOs.
  • seq_cust_notif_task_annots sequence increment increased to 50 in SQL Server

Notes​

  • None.

[6.7.0]​

Breaking Changes​

  • None.

New Features​

  • Added support to compare notification tasks with customers generated via isolation points against current state of network.
  • Added periodic task support for BulkNarOperation with 'runEverySeconds' set.
  • Replaced fixed delivery methods with values from the database and connected them to access types.
  • Added getDownstreamEquipmentTree and findUpstreamEquipment GraphQL APIs and routes to provide device tree hierarchy and feeder info (through findUpstreamEquipment).
  • EDNAR can now send webhook notifications (including query data) to registered end points. Webhook registration is controlled via the new webhooks config APIs:
    • New GET /webhooks config API endpoint for reading current webhook registrations.
    • New POST /webhooks config API endpoint for registering a new webhook.
    • New PUT /webhooks/<ID> config API endpoint for updating an existing webhook registration.
    • New DELETE /webhooks/<ID> config API endpoint for deregistration of a webhook.
  • Added Custom Data Keys and Custom Data Values to represent arbitrary key/value data that can now be stored against a NAR
    • Added new GET, POST, PUT and DELETE endpoints at /api/v0/config/custom-data-keys/ to allow administrators to manage the available set of keys
    • Added getCustomDataKeys query to GraphQL to allow users to query for the available set of keys
    • Added new GET, PUT and DELETE endpoints at /api/v0/nars/:narId/custom-data/ to allow users to manage the values stored against custom keys for a specific NAR
    • Added customDataValues field to the Nar in GraphQL to allow users to query the values stored against custom keys for a specific NAR
    • New database tables custom_data_keys and nar_custom_data_values created to store this information
    • New audit log action MODIFY_CUSTOM_DATA_VALUE is recorded for when a custom data value is modified directly on a NAR (but not as part of a complete NAR save)
    • New notification action MODIFY_CUSTOM_DATA_VALUE can trigger notifications for when a custom data value is modified directly on a NAR (but not as part of a complete NAR save)

Enhancements​

  • Added ENABLE_DEVICE_TREE_IN_MAP app option (default false) to show/hide device tree in the map.
  • Added new tracedApparatusSelection boolean field to Access Authorities. Used by frontend when selecting and displaying apparatus to be covered.
  • Added log entries when database.startupScripts is either not present in the configuration, or contains no scripts to run.
  • New field supplyPoints under CustomerNotificationTask in the GraphQL API, allowing the retrieval of a list of supply point (EnergyConsumer) IDs for a given task.

Fixes​

  • Configuration option database.startupScripts now supports both \ and / as path separators on Windows.

Notes​

  • None.

[1.7.0]​

Added​

System/Broadcast Notifications​

  • System Notifications added, as well as commands, queries, and API endpoints to fetch, mark, delete and add new notifications.

Generator Pages​

  • Added generator pages to the nar form.

NAR Notifications​

  • Now all blank spaces are removed from phone numbers before they are used to send an SMS.

Comments​

  • Added support to add a pinned comment.
  • Added support to fetch pinned comments.

Admin​

  • Added support to fetch privileges.
  • Added support to fetch, add, edit and delete roles.
  • Added support to add and delete users from roles.

PoN​

  • Added query parameter 'requestNewId' to the setPonProject endpoint. This query parameter allows for the current PoN project present in a NAR to be overridden by another (example: When the project has been cleared in the client and a new project is created for that NAR).

Security​

  • Added appropriate cache control headers to api responses.
  • Secure flag is now added to cookies if ssl is enabled.
  • Added best practices security headers to all HTTP responses.
  • HTTPS protocols can be configured. Currently supporting TLS 1.1 and TLS 1.2. Default configuration is TLS 1.2 only.

Fixed​

Notifications​

  • Fixed bug that caused the isHtml flag in notification templates to be ignored.
  • Fixed bug that would prevent an email from being sent if the dmsId in the nar was null and it was used in the notification template for said email.

Customers Endpoint​

  • The get-customers endpoint now returns a 200 OK response for the case where an isolation trace returned no NMIs.

DMS Integration​

  • PoN Interruptions dates and times are now sent to DMS.

Users​

  • 'System' users are now not returned from user queries.

Changed​

Notifications​

  • DMS_REVOKED notification has been split into more fine grained notifications: DMS_REVOKED_SUBMITTED, DMS_REVOKED_PRECHECKED, DMS_REVOKED_APPROVED.

Questions​

  • Questions are now paginated on the server. In addition, the API now accepts a search term to filter the returned list of questions.

[1.6.0]​

Added​

Notifications​

  • Interested parties can now be configured to receive notifications.

Changed​

NAR Search​

  • Increased maximum page size for NAR Search API request from 100 to 10000

DMS Integration​

  • When cancelling a NAR, all NAR fields are now updated in the DMS to ensure DMS is in sync with the NAR.
  • Planned interruption text send to DMS has been updated to new format.
  • When saving or cancelling a NAR with an associated DMS job, the API now includes flags if the DMS failed to be updated.

Fixed​

Nar Copy​

  • Fixed bug that caused nar-copies to fail if the nar contained a disabled user. Now disabled users get filtered out of the copy.

Lead-Time​

  • Fixed leadTime calculation to use the correct reference date which should be the latest SUBMIT date or today if the nar is in a DRAFT or PENDING_SUBMISSION state.

Changed​

Nar Report​

  • Increased request timeout for NAR report to 60 seconds.

PoN API​

  • Updated PoN service to be in sync with the PoN API after the changes.

PoN Project​

  • Moved the checkedOutageViewer, siteCheckCompleted and gisChangeRequired values back to the PoN project from the PoN interruptions.

[1.5.0]​

Added​

  • Added Interested Parties to the EDNAR server - these are external parties whose information may need to be associated with a NAR. NAR fetching and saving endpoints are updated to expect Interested Parties in the NAR body.

Added​

  • Updated the nar-copy endpoint to accept an optional body that allows for the creation of multiple copies of a nar based on the criteria contained in the body.
  • Added the following app-options [EXTERNAL_MAP_ASSET_VIEW_URL, EXTERNAL_MAP_OUTAGE_VIEW_URL] used to configure the external map urls.

Fixed​

PoN Integration​

  • Fixed url to order-cancellation-notices.

NAR audit-log​

  • Added missing value to the audit log entry for a NAR update that puts a NAR within lead time.

  • Boolean NAR questions can now be configured to enable required attachments when answered either yes or no.

Changed​

  • The nar-status now includes the PoN project status if there is a PoN project associated with the NAR.

Added​

  • Added query parameter "includeTasks" to the get-pon-project endpoint to select whether to include the tasks with the PoN project information payload.

  • Privileges to assign planners in all states from submitted (inclusive) except terminal states.

  • Fixed date comparisons not using a "day" level granularity
  • Fixed calculation of calendar days based on number of business days
  • Fixed bug where an update to DMS switching details could cause a null pointer exception.
  • Clash detection results now exclude the NAR the request was made for.
  • Requesting PON cancel order config now returns 404 if it is not configured.
