API Clients

This page will walk through how to create and use API clients in EDNAR for machine-to-machine authentication. To create and manage API clients, you will need the ADMIN_API_CLIENTS privilege.

Creating API Clients​

In the EDNAR Admin app, go to Configuration -> API Client Management

Click "Create API Client"

Give the API Client a description and assign it a role (the role will dictate what privileges the client has)

Click "Save". You will be presented a Client Secret similar to this:

Important

You must copy the Client Secret and save it somewhere secure now - you won't be able to access it again!

Back on the API Client form, you will now see a Client ID, which you can also copy by clicking the dotted box.

Using API Clients for authentication​

To use your Client ID and Client Secret to authenticate with EDNAR, we use an OAuth2 flow. The easiest one is the OAuth2 Client Credentials flow.

In your preferred HTTP Client, configure Auth, and select OAuth2 from the list of available options.

Fill in "Grant Type" as "Client Credentials" for this example.

For access token URL, use the host where the EDNAR App Server is running, with the endpoint /api/token. Use your Client ID as the Client ID, and your Client Secret as the Client Secret:

Click "Fetch Token" and it should issue you an Access Token:

Manually perform the OAuth2 Client Credentials flow rather than use a HTTP Client, you can follow the spec for a request here: https://datatracker.ietf.org/doc/html/rfc6749#section-1.3.4

To do so, you need to make a POST request to /api/token with the following body encoded in x-www-form-urlencoded format with the following parameters:

grant_type=client_credentials&
client_id=<YOUR_CLIENT_ID>&
client_secret=<YOUR_CLIENT_SECRET>

or,

grant_type=client_credentials

and then include an Authorization header with the Client ID and Client Secret encoded as Basic auth credentials: https://en.wikipedia.org/wiki/Basic_access_authentication

The response will contain an Access Token.

Either way, you can then use that Access Token as a Bearer token for subsequent EDNAR requests, without needing to include a token in the cookie or an X-XSRF-Token header. You are now logged in as an API Client!

The token will expire at some point, so you'll need to repeat the request to /api/token every time it expires. If you'd rather refresh the token using a refresh token, you can use the more complex OAuth2 Authorization Code flow described here: https://datatracker.ietf.org/doc/html/rfc6749#section-1.3.1 with /api/authorize as the Authorization Code endpoint and /api/token as the Access Token endpoint.
