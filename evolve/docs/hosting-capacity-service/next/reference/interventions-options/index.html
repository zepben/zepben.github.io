Intervention List

TARIFF_REFORM and CONTROLLED_LOAD_HOT_WATER​

Tariff Reform and Controlled Load Hot Water are implemented identically and reference the same database tables.

Overview​

These interventions modify the time of use patterns of customer electricity loads. They work by applying load reshape profiles to eligible customers based on specified criteria.

Tables referenced:

  • load_reshape_strategies - Defines targeting criteria and application rules, which then references:
    • load_reshape_instances - Contains the actual load modification profiles

Configuration Example​

InterventionConfig(
    base_work_package_id="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    year_range=YearRange(2026, 2030),  # Interventions should be applied for a range of years without gaps 
    intervention_type=InterventionClass.TARIFF_REFORM # or InterventionClass.CONTROLLED_LOAD_HOT_WATER
    allocation_criteria="load_reshape_strategy_1"  # Must match a criteria_name in load_reshape_strategies table
)

Database Tables​

load_reshape_strategies​

This table defines the rules for selecting and applying load reshape strategies to customers. Strategies are applied in order of rank to matching customers (i.e. 1 gets applied first, then 2, etc.). A customer can only receive one load reshape, so once they have been assigned a strategy they are excluded from subsequent strategies. All specified filters must match for a customer to be eligible. NULL/omitted filters are ignored. Some filters will depend on data being available in the customer data, such as customer_type, has_controlled_load, and has_controlled_load.

link to load_reshape_strategies TABLE

load_reshape_instances​

This table contains the actual load modification profiles that are applied to customers.

LINK TO load_reshape_instances TABLE

Profile Modes:

  • ADDITIVE: Each value (in W) is added to the load at the corresponding time interval, e.g {100, 50, -75, ...} adds 100W in interval 1, 50W in interval 2, subtracts 75W in interval 3, etc.
  • MULTIPLICATIVE: Each value (as a proportion) is multiplied by the load at the corresponding time interval
    • Example: {1.0, 0.5, 1.2, ...} keeps load unchanged in interval 1, halves it in interval 2, increases by 20% in interval 3, etc.

COMMUNITY_BESS​

Community Battery Energy Storage Systems (BESS), also called Neighborhood BESS, are mid-sized, front of the meter batteries that are installed in the low-voltage network to mitigate high load and generation issues, espcially high solar penitration or very 'spikey' loads. This intervention addresses voltage and thermal constraint violations by strategically deploying batteries at identified problem locations.

Overview​

The COMMUNITY_BESS intervention identifies candidate locations where network metrics exceed defined thresholds, then ranks and allocates battery installations based on the severity of constraints. Candidates are prioritized by the combined sum of:

  • gen_exceeding_normal_voltage_cecv (generation causing voltage violations)
  • load_exceeding_normal_thermal_voltage_vcr (load causing thermal/voltage violations)

This sum is calculated across all applicable years for each candidate location.

Tables referenced:

  • intervention_candidate_criteria - Defines threshold conditions for identifying problem locations
  • intervention_candidates - Stores identified candidates with their rankings
  • bess_allocation_criteria - Specifies battery placement and sizing rules
  • bess_instances - Contains battery technical specifications and control parameters
  • bess_profiles - Battery operational profiles

Configuration Example​

InterventionConfig(
    base_work_package_id="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",  # either specifided or generated in code, eg `base_work_package_id = base_work_package[id]`
    allocation_limit_per_year=100,  # Maximum number of batteries to install per year
    intervention_type=InterventionClass.COMMUNITY_BESS,
    candidate_generation=CandidateGenerationConfig(
        type=CandidateGenerationType.CRITERIA,  # Required for COMMUNITY_BESS
        intervention_criteria_name="bess_threshold_set_1" # References intervention_candidate_criteria.name
    ),
    allocation_criteria="bess_allocation_criteria_1",  # References bess_allocation_criteria.name
    specific_allocation_instance="bess_instance_1",    # Optional. References bess_instances.name, if specified, only this battery type is used, otherwise, the smallest viable battery is selected
)

LV_STATCOMS​

Low Voltage Static Synchronous Compensators (LV STATCOMs) are voltage regulating devices installed on LV networks to provide dynamic reactive power compensation. These power electronics-based devices act as either a source or sink of reactive AC power, rapidly responding to voltage fluctuations to maintain network stability.

Overview​

Candidates are identified where network metrics exceed defined thresholds, then ranked and allocated based on the severity of constraints. Candidates are prioritized by the combined sum of:

  • gen_exceeding_normal_voltage_cecv (generation causing voltage violations)
  • load_exceeding_normal_thermal_voltage_vcr (load causing thermal/voltage violations)

This sum is calculated across all applicable years for each candidate location.

Tables referenced:

  • intervention_candidate_criteria - Defines threshold conditions for identifying problem locations
  • intervention_candidates - Stores identified candidates with their rankings
  • lv_statcom_allocation_criteria - Specifies STATCOM placement rules and network requirements
  • lv_statcom_instances - Contains STATCOM technical specifications and control parameters

Configuration Example​

InterventionConfig(
    base_work_package_id="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",  # Used by PRRP to query enhanced metrics
    year_range=YearRange(
        min_year=2026,  # Earliest year to identify and apply intervention candidates
        max_year=2030   # Latest year to identify and apply intervention candidates
    ),
    allocation_limit_per_year=100,  # Maximum number of LV STATCOMs to install per year
    intervention_type=InterventionClass.LV_STATCOMS,
    candidate_generation=CandidateGenerationConfig(
        type=CandidateGenerationType.CRITERIA,  # Required for LV_STATCOMS, references intervention_candidate_criteria.name
        # Defines thresholds (e.g., min_voltage_outside_limits_hours), measurement zones exceeding ALL specified thresholds become candidates
        intervention_criteria_name="threshold_set_2"
    ),
    allocation_criteria="lv_statcom_allocation_criteria_1",  # References lv_statcom_allocation_criteria.name
    specific_allocation_instance="lv_statcom_instance_1",    # Optional: references lv_statcom_instances.name
                                                              # If specified, only this STATCOM type is used
                                                              # If omitted, first suitable instance (matching year/feeder) is selected
)

lv_statcom_allocation_criteria​

This table defines rules for STATCOM placement and network compatibility requirements.

Column NameData TypeConstraintsDescription
nameTEXTPRIMARY KEY, NOT NULLUnique identifier for this STATCOM allocation rule set
percentage_distance_from_measurement_zone_terminalREALNOT NULLDistance from measurement zone terminal, as a percentage of total zone length, where STATCOM is sited
minimum_x_r_ratioREALOptionalMinimum required reactance-to-resistance (X/R) ratio for network lines between the measurement zone terminal and STATCOM placement location. Ensures adequate network inductance for effective reactive power compensation

Network Compatibility: The minimum_x_r_ratio filter ensures STATCOMs are only deployed where network characteristics allow effective reactive power support. Networks with insufficient inductance (low X/R ratio) may not benefit from STATCOM installation.

lv_statcom_instances​

This table contains the technical specifications and control parameters for STATCOM devices. Each instance defines a specific STATCOM configuration that can be deployed.

Column NameData TypeConstraintsDescription
nameTEXTPRIMARY KEY, NOT NULLUnique identifier for this STATCOM configuration
kvaREALNOT NULLRated apparent power capacity in kVA
voltage_curvex_refTEXTNOT NULL, voltage_curvex_refReference voltage method for determining monitored voltage in per-unit. Options include: rated voltage of controlled element, average voltage over time window, or combination
var_change_toleranceREALNOT NULLAcceptable difference between desired and actual reactive power before control adjustments are made. Prevents excessive control iterations and stabilizes operation
voltage_change_toleranceREALNOT NULLMinimum voltage change threshold to trigger control action. Prevents unnecessary responses to minor fluctuations
ref_reactive_powerTEXTNOT NULL, ref_reactive_powerBaseline reference for reactive power control. Currently set to VARMAX, limiting provided/absorbed reactive power by kvarMax and kvarMaxAbs
hysteresis_offsetREALOptionalDead-band offset in control response. Shifts the control curve to prevent small voltage fluctuations from triggering immediate reactive power adjustments
avg_window_lengthINTEGERNOT NULLDuration (in time intervals) over which voltage values are averaged. Smooths fluctuations for more stable control
nptsINTEGERNOT NULLNumber of data points in the voltage-reactive power control curve
x_arrayTEXTNOT NULL, DOUBLE ARRAYX-axis values for control curve (per-unit voltages)
y_arrayTEXTNOT NULL, DOUBLE ARRAYY-axis values for control curve (percentage of reactive power support)
phaseINTEGERNOT NULLNumber of phases for the STATCOM: 1 (single-phase) or 3 (three-phase)

STATCOM Selection:

  • If specific_allocation_instance is specified in the config, only that STATCOM instance is used
  • Otherwise, the first instance matching the required year and feeder from the database query is selected

Voltage-Reactive Power Curve: The x_array and y_array define the volt-var characteristic curve that determines how the STATCOM adjusts reactive power output in response to measured voltage. This curve is the primary control mechanism for voltage regulation.

Control Stability: The tolerance and hysteresis parameters work together to prevent control oscillation:

  • var_change_tolerance prevents continuous small adjustments to reactive power
  • voltage_change_tolerance ignores minor voltage variations
  • hysteresis_offset creates a dead-band preventing rapid switching between control states
  • avg_window_length smooths voltage measurements over time
