How to Run a Model Calibration Work Package

1. Introduction to Calibration​

Accurate network model calibration is a prerequisite for reliable power system analysis particularly crucial for voltage regulation studies and determining optimal transformer tap settings. The objective is to ensure that the power flow models generated are an accurate reflection of real-world system conditions, providing a validated baseline for further analysis.

For more information on why network calibration is important and the underlying concepts, see What is Network Model Calibration and why is it useful?

2. Calibration Methodology​

The calibration process integrates measured field data with the Hosting Capacity Modules OpenDSS power flow simulation environment through the following steps:

2.1. Data Ingestion and Preparation​

Required inputs (these should have been automated into your deployment during Hosting Capacity Module deployment):

  • Historic watts, vars, and voltages OR historic amps, power factor, and voltages at the energy consumer level (i.e., supply point or NMI level). This is often referred to in the industry as PQV or Power Quality Data. These datasets do not need to cover complete weeks, months, or years; rather, they should cover a range of grid conditions, including minimum, maximum, and typical network loading periods. Examples include selecting 7 am, 12 pm, 7 pm, and 3 am to give insights during periods of medium demand, high generation, high demand, and low demand.
  • Feeder head voltages that are mapped to the feeder identifier used within the Energy Workbench. These voltages can be the phase-to-phase voltages measured at the MV bus of the feeder's Zone Substation. This input is important to remove the uncertainty for the upstream HV network and the modelled voltage regulation settings from the model. This data is stored within the feeder_head_metrics table.

2.2. Power Flow Model - Calibration 'Mode'​

When calibration work packages are run, the OpenDSS power-flow models are built to model a specific historic timestamp for which historic input data exists:

  • The net demand position of each usage point (NMI) in the network is modelled as a 'bi-directional' Load object within the model, which maps directly to the instantaneous 5-minute P and Q power data stored within the TimescaleDB time-series database.
  • The feeder head tap changer representation is replaced with an ideal voltage source, its value set to the measured actuals from the feeder_head_metrics table.
  • measurement_zone objects are strategically allocated to the feeder head and the usage points at each customer connection (NMI) to provide voltage measurements that align with the location of real-world smart meter voltage actuals, facilitating comparison between modelled and measured values.

2.3. Load Flow Execution​

  • A load flow analysis is executed on the updated OpenDSS power flow model for each defined time case within the calibration work package.
  • Results are streamed to the specific energy_meter_voltages_raw table, where the conducting_equipment_mrid lines up with the energyconsumer_mrid (supply point) and can then be used as part of the automated analysis to infer off-load tap changer positions and 'calibrate' the model.

2.4. (Optional) Off-Load Tap Position Determination​

  • For each energyconsumer location, a voltage difference (ΔV) is calculated based on the difference between the average of the available real-world measured phase voltages and the average of the modelled phase voltages.
  • This voltage delta is then used in combination with the associated transformer's current tap position and tap size to determine an adjustment, expressed in terms of the number of taps the modelled position should be moved from the current 'as-built' modelled position.
  • The configuration for triggering tap adjustments is defined within the calibration work package settings. Users can specify whether off-load tap adjustments should be performed by enabling or disabling the perform_tap_adjustments flag in the configuration file. By default, tap adjustments are performed unless explicitly disabled. Provide more details about the configuration used to trigger this process. Clarify whether tap adjustments are always performed. This allows flexibility for users to decide whether to include tap adjustments based on the specific requirements of their calibration study.
  • The resulting tap positions are stored in the calibrated_taps table, where various 'sets' of off-load tap positions can be determined for different calibration studies. These sets of off-load tap positions can then be selected by users on demand as part of the work package configuration when running future time-series modelling work packages.

2.5. Model Validation and Discrepancy Analysis​

  • Similar to the Off-Load Tap Position Determination flow, this workflow can be used to compare the voltages (V) obtained from the load flow simulation with the actual measured voltages (i.e., the raw PQV data) at energyconsumer locations. The voltage difference (ΔV) is calculated for these points: ΔV = Actual Raw Voltages - Load Flow Voltages.
  • Review of these energyconsumer level voltage delta results, without seeking to adjust off-load tap positions, provides an ability to measure the accuracy of the model and develop a view for the range of model uncertainty for various network demand scenarios.
  • Further analysis of the relationship between demand and voltage error can provide insights into the uncertainty of the impedance model; however, no built-in tools are included for this analysis.

  • Processed results are stored within the calibration_meter_results table, which contains voltage delta calculations and flagged instances for further analysis.
  • Processed results are stored within the calibration_meter_results table.
  • Instances where ΔV exceeds a predefined threshold (e.g., 1.5%) are flagged.
  • A list of transformers and associated energyconsumers exhibiting these significant voltage deviations is generated for review and further investigation. This step is crucial for identifying potential model inaccuracies or data anomalies.
  • For both the model validation use case and the off-load tap position use case, users are able to configure thresholds that determine if a measured actual is valid and should be included, or if a simulated voltage is valid and should be included in the calculation of voltage deltas. For example, users can specify a threshold of ±1.5% for voltage deviations, meaning that any measured or simulated voltage outside this range will be excluded from the analysis. These thresholds are typically defined in the calibration work package configuration file under the voltage_threshold parameter, allowing flexibility to adapt to different network conditions and study requirements.

3. Subsequent Application: Utilizing Calibration Outcomes​

Following successful calibration, the validated Energy Workbench model and its inferred parameters (such as off-load tap positions) serve as the analytical basis for future applications.

The inferred tap position information can be called directly by users as part of a work package within the Hosting Capacity Module or alternatively customers can incorporate these results into their source systems and have these inferred tap positions used by the Energy Workbench, and all its modules, such as the PowerFactory or Sincal model builders.

Prerequisites

See the How to Run a Work Package guide for prerequisites, which are the same for running a model calibration work package.

How to run

Option 1: Using GraphQL​

Run a GraphQL query to start a model calibration work package. 

mutation {runCalibration(calibrationName: "test999", calibrationTimeLocal: "2025-01-15T00:00:00" )}

This will start a model calibration work package with the specified name and local time, for all feeders in the network model.

Option 2: Using the Python API​

in the eas-client-python client, run:

run_hosting_capacity_calibration(`calibration_name`, `local_calibration_time`)

This will start a model calibration work package with the specified name and local time, for all feeders in the network model.

Where calibrationName is a unique identifier for the calibration, and calibrationTimeLocal is the local time of the calibration in ISO 8601 format (it should not have a timezone offset, as the calibration time is assumed to be in the local timezone of the network model, which is set up in the underlying model).

Using the results​

  1. Tap settings
  2. Raw results in the database.
