PowerFactory Model Builder - Backend Configuration

This will be of limited interest to most users. It is primarily intended for developers who want to integrate with or extend the PowerFactory Model Builder backend services.

Starting​

These instructions will allow you to get a copy of the project running on your local machine for development and testing purposes and/or running a docker container as well.

Maven​

Add to your pom.xml:

    <dependency>
        <groupId>com.zepben.powerfactory</groupId>
        <artifactId>exporter</artifactId>
        <version>0.7.0</version>
    </dependency>

Usage​

For a basic example of using the PowerFactory exporter refer to the demonstration code in file com.zepben.powerfactory.exporter.demo.Demo.kt

val logger: Logger = LoggerFactory.getLogger("main")

GrpcChannelFactory.create(ConnectionConfig(demoConfig.ewb.host, demoConfig.ewb.grpcPort, demoConfig.ewb.secureConnection)).use \{ channel ->
    val dgsGenerator = DgsGenerator(DgsGenerator.Config(), XlsxDgsWriter()) \{ NetworkConsumerClient(channel) \}

    when (val result = dgsGenerator.generate(demoConfig.generationConfig)) \{
        is DgsGenerationResult.Success<XSSFWorkbook> -> \{
            writeFile(result.dgs, "output/model.xlsx")
        \}

        is DgsGenerationResult.Failure<*> -> \{
            logger.error("Failed to create model with the following errors:")
            result.errors.forEach \{ (m, t) -> logger.error(m, t) \}
        \}
    \}
\}

Documentation 📦​

Workflow Overview​

  1. User Request PowerFactory Model by sending Model Config to PowerFactory Model Builder
  2. Model Config sent to PowerFactory Exporter
    This library starts here
  3. Request Network Model from Energy Work Bench
  4. Receive Network Model from Energy Work Bench
  5. Return DGS to PowerFactory Model Builder
    This library ends here
  6. Sent DGS to PowerFactory Template Service
  7. Return DGS to PowerFactory Model Builder
  8. Return DGS to EWB App Server

EWB Config​

VarDescriptionExample
hostHost for the energy-workbench-server'127.0.0.1'
grpc_portPort for the energy-workbench-server is listening on for grpc requests9014
secureIndicates if TLS should be enabled for the connection with the energy-workbench-servertrue
caFilenameFilename of CA to use when verifying certificate of the energy-workbench-server. Leave unspecified to use system CAs.'./ewb_ca.cert'
authConfig for the authentication used for the connection with energy-workbench-serverAuth Config

Notes:

  • caFilename and auth have no effect if secure is set to false.

Auth Config​

VarDescriptionExample
clientIdThe client ID of the OAuth application to authenticate with'notARealClientID'
clientSecretThe client secret of the OAuth application to authenticate with'notARealClientSecret'
usernameUsername for a user of the OAuth application to authenticate with'notARealUsername'
passwordPassword for a user of the OAuth application to authenticate with'Swordfish'
confAddressURL to fetch certain token request fields from'127.0.0.1:9000/ewb/auth'
audienceIdentifier for set of APIs to request access for'https://evolve-ewb/'
issuerDomainDomain name of OAuth token provider'zepben-dev.au.auth0.com'
authMethodType of authentication API to use. Currently, this should be left unspecified.AuthMethod.OAuth
confCAFilename of CA to use when verifying certificate of confAddress. Leave unspecified to use system CAs.'./conf_ca.cert'
authCAFilename of CA to use when verifying certificate of issuerDomain. Leave unspecified to use system CAs.'./auth_ca.cert'

Notes:

  • There are two OAuth grant types supported: password and client credentials.
    • To use a password grant, specify username and password. Ensure clientId refers to an application that supports password grants.
    • To use a client credentials grant, specify clientSecret. Ensure clientId refers to an application that supports client credential grants.
      • To use a client credentials grant with managed identity (EntraID only), don't specify clientSecret. Ensure clientId refers to an application that supports client credential grants.
  • There are two ways of specifying the token audience and the domain of the token issuer:
    • Specify confAddress to point to a json file served over HTTPS. If specified, confCA is used to verify the certificate.
    • Specify audience and issuerDomain directly.

Feeder Colours Specification File​

This file is used to control the colours assigned to each feeder in the resulting DGS file. It is a json file that contains a mapping from feeder mrids to the colour number to assign for said feeder.

Example:

{
  "feeder1" : 1,
  "feeder2" : 2,
  "feeder3" : 3
}

Feeder Max Load Specification File​

This file is used when spread feeder max is set to true on model generation. Instead of linking load data to a database, a static value is assigned to them base off maximum load data provided.

HV Customer Max Demand Specification File​

This file is used to populate load data for HV customers.

NMI List Specification File​

This file provides the full list of customers with time series data in the database.
